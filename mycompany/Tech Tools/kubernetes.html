Roadmap

Learn Kubernetes in the right order so it actually sticks

Most people bounce off Kubernetes because they start with syntax instead of intent. The right path is: understand the reconciliation model, map the core objects, watch traffic and storage behave, then practice commands and failures until the cluster stops feeling magical and starts feeling predictable.

01

Foundation

Kubernetes is a system for keeping declared state true. Learn Pods, Nodes, the API server, and the control loop before anything else.

02

Workloads

Understand when to use a Deployment, StatefulSet, DaemonSet, Job, and CronJob so you pick the right abstraction for the app.

03

Platform

Learn Service types, Ingress, DNS, ConfigMaps, Secrets, PersistentVolumeClaims, and namespaces. This is where apps become platforms.

04

Operations

Practice scaling, rollouts, rollbacks, probes, logs, events, port-forwarding, and basic node maintenance until they feel routine.

05

Production thinking

Add resource requests, limits, RBAC, network policy, autoscaling, and cleanup discipline so experiments do not turn into mystery debt.

The shortest honest definition of Kubernetes is: a distributed control system that keeps your application fleet close to the state you declared.

You describe intent

Usually in YAML or through kubectl. You say what should exist, not how every step should be executed.

Controllers compare reality to intent

If a Pod dies, the Deployment notices the count is wrong and asks for a replacement.

Stable interfaces hide churn

Services, DNS names, and PVCs stay predictable while Pods restart, move, and update.

When Kubernetes is worth it

Use it when you need repeatable deployments, multiple services, rolling updates, self-healing, or consistent runtime behavior across environments.

Not everything needs Kubernetes. A single internal app may be easier with Docker Compose, serverless, or a managed PaaS.
Mental model

See the whole flow before you zoom into YAML

Every Kubernetes topic gets easier once you can picture how a request moves through the cluster. A deployment is not just a file. It is a change to desired state that gets stored, scheduled, executed on nodes, observed, and corrected over time.

1 You write a manifest or run a kubectl command.
2 The API server validates and accepts the object.
3 etcd stores the new desired state.
4 A controller notices what should exist.
5 The scheduler picks a worker node.
6 kubelet starts the Pod through the container runtime.
7 Services and DNS make the Pod reachable by a stable name.
Architecture

Know what each major component is responsible for

When a cluster misbehaves, you debug faster if you know which layer owns the problem. Scheduling, networking, storage, and application health are related, but they are not the same subsystem.

Control plane components

API Server

The entry point for kubectl, controllers, and automation. If the API server is unhealthy, the cluster becomes hard to operate.

etcd

The distributed key-value database for cluster state. Backups matter because etcd is where intent and metadata live.

Scheduler

Chooses which node gets each unscheduled Pod based on resources, affinity, taints, and policy.

Controller Manager

Runs the loops that make Deployments, Jobs, StatefulSets, and Nodes converge toward the desired state.

Worker node components

kubelet

The node agent. It watches for assigned Pods and makes sure containers really start and stay healthy.

Container runtime

Actually pulls images and runs containers. Think containerd or another CRI-compatible runtime.

kube-proxy or equivalent

Implements Service networking rules so traffic can reach the right backend Pods.

CNI and CSI plugins

CNI handles Pod networking. CSI handles storage attachment and provisioning. They are extensions, not built-in magic.

Objects and workloads

Choose the right Kubernetes object for the job

Kubernetes feels bloated until you realize each object solves a different kind of runtime problem. The real skill is not memorizing every field. It is picking the right abstraction so the platform can help instead of fight you.

Object Use it for Common mistake
Pod A single runtime unit with one or more tightly coupled containers. Running bare Pods for production apps instead of using a controller.
Deployment Stateless web apps, APIs, and anything you want to scale and roll out safely. Expecting stable Pod names or stable storage identity.
StatefulSet Databases, queues, and systems that need stable identity or persistent storage. Using a Deployment for a stateful system and losing predictable identity.
DaemonSet One Pod per node, such as log shippers, security agents, or metrics collectors. Treating it like a normal horizontally scaled service.
Job One-time batch work that should finish and stop. Using it for something that should run forever.
CronJob Scheduled batch work like backups, reports, or cleanups. Forgetting concurrency policy and letting overlapping jobs pile up.
Service A stable virtual endpoint in front of changing Pod IPs. Trying to connect directly to Pod IPs from clients.
Ingress HTTP and HTTPS routing into the cluster. Assuming it works by itself without an ingress controller.
ConfigMap Non-secret configuration such as feature flags or environment defaults. Storing passwords in plain config.
Secret Sensitive values that should be handled separately from regular config. Thinking base64 encoding equals encryption or strong security.
PVC Requesting persistent storage for a workload. Assuming storage exists without a StorageClass or provisioner.
Namespace Logical isolation for teams, apps, or environments inside one cluster. Using namespaces as a complete security boundary by themselves.
Platform pillars

Networking, storage, security, and operations are where clusters become useful

Kubernetes is not just a scheduler. It is a platform surface. The four pillars below are where most production value and most production confusion both come from.

Networking

Every Pod gets an IP. Services give stable access. DNS names let one service call another. Ingress manages HTTP entry from outside.

  • ClusterIP is internal-only traffic.
  • NodePort exposes a port on each node.
  • LoadBalancer asks the platform for an external balancer.
  • NetworkPolicy limits who can talk to whom.
Start here: kubectl get svc,endpoints,ingress -n app

Storage

Container filesystems are ephemeral. If the Pod dies, that writable layer dies too. Use persistent volumes when data must survive restarts.

  • emptyDir lives only for the life of the Pod.
  • PVC requests durable storage.
  • StorageClass controls how volumes get provisioned.
  • Stateful apps usually need stable storage and stable identity.
Start here: kubectl get pvc,pv -n app

Security

Security in Kubernetes is layered. RBAC controls who can do what. ServiceAccounts define workload identity. SecurityContext constrains process behavior.

  • Run with least privilege.
  • Avoid mounting secrets you do not need.
  • Prefer read-only filesystems where possible.
  • Use namespaces and network policy together, not alone.
Start here: kubectl auth can-i get pods -n app

Operations

Production clusters reward operational discipline. Requests and limits influence scheduling. Probes govern traffic and restart behavior. Rollouts reduce risk.

  • Readiness controls traffic eligibility.
  • Liveness controls restart behavior.
  • kubectl rollout turns upgrades into a managed workflow.
  • Events and logs are your first responders.
Start here: kubectl describe pod <pod> -n app
Manifest thinking

Read YAML like the cluster reads it

A strong Kubernetes manifest says what should exist, how it should identify itself, how it should be reached, and what healthy runtime behavior looks like. This example is a good baseline for a stateless web service.

Annotated starter manifest deployment + service 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: web
          image: nginx:1.27
          ports:
            - containerPort: 80
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 3
            periodSeconds: 5
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "256Mi"
---
apiVersion: v1
kind: Service
metadata:
  name: web
  namespace: app
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
  type: ClusterIP
metadata

Name and namespace tell Kubernetes where this object lives. Labels become critical later because Services and selectors depend on them.

replicas + selector

The Deployment promises the replica count. The selector must match the Pod template labels or the controller cannot manage what it creates.

readinessProbe

This is how you stop broken Pods from receiving traffic. Without readiness, your rollout can look healthy while users still fail.

resources

Requests influence scheduling. Limits cap usage. Defining neither is one of the fastest ways to create noisy-neighbor problems.

service selector

The Service sends traffic to Pods with app: web. If labels drift, the service can exist and still have zero usable endpoints.

Learn faster

Use kubectl explain deployment.spec.template.spec.containers whenever a YAML field feels mysterious. It is one of the best built-in teachers.

Command atlas

The kubectl and kind cheat sheet you will actually use

Search by keyword or filter by category. The point is not just to list commands, but to explain what each one is for so you can reach for the right tool quickly under pressure.

33 rows
Command What it does When to use it
kind create cluster --name k8s-lab Creates a disposable local cluster with the name k8s-lab. Best starting point for laptop practice if you want a clean reset path.
kubectl cluster-info Shows the control plane endpoints and confirms kubectl can reach the cluster. Run it immediately after cluster creation or when you think your context is wrong.
kubectl get nodes -o wide Lists worker nodes and adds extra detail like internal IP and runtime information. Use it to confirm the cluster shape and whether nodes are Ready.
kubectl config get-contexts Shows every cluster context kubectl knows about. Useful when you are juggling local, staging, and production clusters.
kubectl config use-context <name> Switches kubectl to a different cluster context. Use this before every serious change so you do not touch the wrong cluster.
kubectl get ns Lists namespaces. Good for orientation and for checking whether your target namespace exists yet.
kubectl create namespace lab Creates a separate workspace called lab. Use namespaces to keep experiments from mixing with other workloads.
kubectl get pods -A Shows every Pod across every namespace. Great when you do not yet know where the problem lives.
kubectl describe pod <pod> -n <ns> Shows detailed Pod information, conditions, and recent events. First stop for Pending, restart loops, probe failures, and scheduling issues.
kubectl logs deploy/<name> -n <ns> --tail=100 Shows recent logs from one Pod belonging to a Deployment. Use it when an app is starting but behaving strangely.
kubectl logs pod/<pod> -c <container> -f -n <ns> Streams logs from a specific container inside a Pod. Use this for multi-container Pods or when you need live output during a test.
kubectl exec -it deploy/<name> -n <ns> -- sh Opens an interactive shell in one container from the Deployment. Helpful for checking environment variables, DNS, mounted files, or app-local state.
kubectl apply -f app.yaml Creates or updates resources from a manifest file. This is the declarative workhorse for repeatable changes.
kubectl diff -f app.yaml Shows what would change if you applied the manifest. Use it before apply when you want a quick safety check.
kubectl delete -f app.yaml Deletes resources created from a manifest file. Best cleanup option when you used a file to create the workload.
kubectl scale deploy/<name> --replicas=5 -n <ns> Changes the desired replica count of a Deployment. Use it to test horizontal scaling and self-healing behavior.
kubectl rollout status deploy/<name> -n <ns> Watches a rollout until it completes or fails. Use it after changing an image or applying a new Deployment spec.
kubectl rollout history deploy/<name> -n <ns> Shows rollout revisions for a Deployment. Useful when you need to understand what changed before rolling back.
kubectl rollout undo deploy/<name> -n <ns> Rolls a Deployment back to the previous revision. Use it when a new image or config change breaks the app.
kubectl set image deploy/<name> <container>=nginx:1.27 -n <ns> Updates the image for one container in a Deployment. Fast way to trigger and observe a rollout during a lab or emergency fix.
kubectl get svc,endpoints,ingress -n <ns> Lists network entry points and the backend endpoints behind Services. Use it when traffic does not reach your Pods.
kubectl port-forward svc/<name> 8080:80 -n <ns> Maps a Service port to your local machine. Best fast path for testing a cluster service from your browser without exposing it publicly.
kubectl get pvc -n <ns> Shows PersistentVolumeClaims and whether storage was bound successfully. Run this whenever a stateful workload is stuck waiting for storage.
kubectl create configmap app-config --from-literal=MODE=demo -n <ns> Creates a ConfigMap from inline key-value data. Great for labs and for separating config from images.
kubectl create secret generic app-secret --from-literal=API_KEY=demo -n <ns> Creates a Secret from inline key-value data. Use it for sensitive config that should not live in plain manifests.
kubectl get events -A --sort-by=.metadata.creationTimestamp Shows cluster events in chronological order. Excellent for spotting image pulls, scheduling failures, or probe flaps across namespaces.
kubectl top pod -n <ns> Shows CPU and memory usage for Pods if metrics-server is installed. Use it to confirm resource pressure before changing requests or limits.
kubectl explain deployment.spec.template.spec.containers Prints built-in documentation for a field path in the Kubernetes API. One of the best ways to learn the schema without leaving your terminal.
kubectl cordon <node> Marks a node unschedulable so no new Pods land there. Use it before maintenance or while isolating a suspect node.
kubectl drain <node> --ignore-daemonsets --delete-emptydir-data Evicts movable Pods from a node to prepare it for maintenance. Use after cordoning when you need the workloads moved elsewhere.
kubectl uncordon <node> Makes the node schedulable again. Use it after maintenance is complete.
kubectl delete namespace lab Deletes the namespace and everything inside it. The safest large cleanup step for labs that were isolated inside one namespace.
kind delete cluster --name k8s-lab Removes the local kind cluster completely. Use this when you want a full reset and no leftover local cluster state.

Tip: pair kubectl describe, kubectl logs, and kubectl get events before you start guessing. Kubernetes usually tells you what it is unhappy about.

Laptop labs

Safe simulation tests you can run on your own PC

These labs are intentionally disposable. The goal is to build instincts without leaving a mess behind. The examples below assume kind plus kubectl, but the workload commands stay mostly the same if you prefer minikube, k3d, or Docker Desktop.

Recommended

kind

Fast, disposable, cluster-shaped. Excellent for learning and for wiping everything when you are done.

Convenient

Docker Desktop

Good if you already use Docker Desktop and want the simplest local toggle-on experience.

Feature-rich

minikube

Popular for local clusters with addons and a friendly developer workflow.

Lightweight

k3d

Runs lightweight k3s clusters inside Docker and feels great on smaller laptops.

Lab 1

Deploy your first app and reach it locally

Create a Deployment, expose it with a Service, and test it from your browser with port-forward.

10-15 min
RunDeployment + service
kubectl create deployment hello --image=nginxdemos/hello
kubectl expose deployment hello --port=80 --type=ClusterIP
kubectl get pods,svc
kubectl port-forward svc/hello 8080:80

What to notice

Pods get created by the Deployment, but the Service is the stable address. Open http://localhost:8080 while port-forward is running.

Cleanup

Ctrl+C
kubectl delete svc/hello deployment/hello
Lab 2

Scale up and test self-healing

See how the Deployment keeps the actual Pod count aligned with the desired replica count.

10 min
RunScaling + self-healing
kubectl create deployment hello --image=nginxdemos/hello
kubectl scale deployment/hello --replicas=3
kubectl get pods -o wide
kubectl delete pod <one-pod-name>
kubectl rollout status deployment/hello

What to notice

Deleting a Pod does not delete the desired state. The controller notices you are below the target and replaces the Pod.

Cleanup

kubectl delete deployment/hello
Lab 3

Force a bad rollout, then recover with rollback

Practice the exact moment that makes Kubernetes feel valuable: safe recovery after a broken image change.

15 min
RunRollout + rollback
kubectl create deployment hello --image=nginxdemos/hello
kubectl set image deployment/hello hello=nginx:1.27
kubectl rollout status deployment/hello
kubectl rollout history deployment/hello
kubectl set image deployment/hello hello=nginx:not-a-real-tag
kubectl get pods
kubectl describe pod <new-pod-name>
kubectl rollout undo deployment/hello
kubectl rollout status deployment/hello

What to notice

The bad image triggers an image pull failure. Kubernetes keeps telling you what happened through Pod status and events, then rollback restores the previous revision.

Cleanup

kubectl delete deployment/hello
Lab 4

Test service discovery and in-cluster DNS

Confirm that service names, not Pod IPs, are the normal way workloads find each other.

10 min
RunDNS and service lookup
kubectl create deployment hello --image=nginxdemos/hello
kubectl expose deployment hello --port=80 --type=ClusterIP
kubectl run dns-check --image=busybox:1.36 -it --rm --restart=Never -- nslookup hello
kubectl run curl-check --image=curlimages/curl:8.7.1 -it --rm --restart=Never -- curl http://hello

What to notice

Service names resolve through cluster DNS. Clients should call hello or the full service DNS name, not raw Pod IPs that can change.

Cleanup

kubectl delete svc/hello deployment/hello
Lab 5

Create ConfigMaps and Secrets, then inspect them

Separate runtime configuration from container images and get comfortable managing both kinds of data.

10 min
RunConfiguration objects
kubectl create configmap app-config --from-literal=APP_MODE=demo
kubectl create secret generic app-secret --from-literal=API_KEY=supersecret
kubectl get configmap app-config -o yaml
kubectl get secret app-secret -o yaml

What to notice

ConfigMaps are for non-sensitive config. Secrets are a separate object type, but remember that base64 is encoding, not strong security by itself.

Cleanup

kubectl delete configmap app-config
kubectl delete secret app-secret
Lab 6

Run a Job and a CronJob like real scheduled work

Get a feel for workloads that finish instead of running forever.

15 min
RunBatch workloads
kubectl create job demo-job --image=busybox:1.36 -- sh -c 'echo start && date && sleep 5 && echo done'
kubectl logs job/demo-job -f
kubectl create cronjob demo-cron --image=busybox:1.36 --schedule="*/5 * * * *" -- sh -c 'date; echo cron tick'
kubectl get jobs,cronjobs

What to notice

Jobs complete and keep history. CronJobs create Jobs on a schedule. That makes them a better fit for backups or reports than a forever-running Deployment.

Cleanup

kubectl delete job demo-job
kubectl delete cronjob demo-cron
Troubleshooting

A simple debugging flow beats panic every time

The mistake most beginners make is jumping straight into random fixes. Start with object state, then events, then logs, then networking, then deeper runtime inspection. The order matters because it cuts guesswork.

60-second debug sequence

1
Check the objects

kubectl get pods,svc,deploy -n app tells you whether the problem is existence, readiness, or routing.

2
Describe before you guess

kubectl describe pod <pod> -n app surfaces scheduling, image pull, probe, and mount errors.

3
Read logs

kubectl logs tells you whether the app crashed, misconfigured itself, or simply never started.

4
Check the network path

Compare Services, endpoints, DNS, and port-forward results before assuming the app is broken.

5
Inspect from inside

kubectl exec helps you verify files, env vars, service reachability, and DNS resolution from the workload itself.

6
Read events for timing

kubectl get events -A --sort-by=.metadata.creationTimestamp often reveals the exact moment things started failing.

Common signals and what they usually mean

Signal Likely meaning First move
Pending The Pod has not been scheduled or cannot mount required resources. kubectl describe pod
CrashLoopBackOff The container starts, fails, and keeps getting restarted. kubectl logs and probe review
ImagePullBackOff The runtime cannot pull the image, often due to a bad tag or auth issue. Check image name and registry credentials
Ready 0/1 The container is running but not healthy enough to receive traffic. Inspect readiness probes and app startup
Service has no endpoints The selector does not match healthy Pods. Compare Service selector with Pod labels
PVC Pending Storage has not been provisioned or bound yet. Check StorageClass and provisioner state
OOMKilled The container exceeded its memory limit. Review memory usage and resource limits

Cleanup

Always leave your local environment clean and boring

Good cleanup is part of learning. If you do not reset your namespaces, contexts, and local clusters, the next lab becomes harder to trust. Reverse the creation order whenever possible and delete at the highest safe boundary.

Fast cleanup checklist

  • Stop any active kubectl port-forward session with Ctrl+C.
  • Delete the namespace if all lab resources live there: kubectl delete namespace lab.
  • Delete file-based resources with kubectl delete -f file.yaml when you used manifests.
  • Reset the active namespace if you changed it: kubectl config set-context --current --namespace=default.
  • Delete the whole local cluster when you want a guaranteed fresh start: kind delete cluster --name k8s-lab.

What not to forget

  • Secrets and ConfigMaps survive until you delete them. They do not disappear just because a Pod did.
  • Stateful workloads may leave PVCs behind even after the app is gone, depending on reclaim policy and storage backend.
  • Namespaces delete asynchronously. Wait until they are actually gone before assuming the cluster is clean.
  • Context mistakes are real. Run kubectl config get-contexts before a destructive command if you work with multiple clusters.
Finish line

Your 0 to hero path after this page

If you can explain the control plane, deploy an app, expose it with a Service, trigger a bad rollout, recover with rollback, and clean up your cluster without fear, you are no longer at zero. The next jump is layering in Helm, CI/CD, ingress controllers, autoscaling, and eventually GitOps.

Next practice targets

  • Add readiness and liveness probes to a real sample app.
  • Mount config through a ConfigMap and sensitive values through a Secret.
  • Run a StatefulSet with a PersistentVolumeClaim and verify data survives Pod replacement.
  • Introduce Helm after raw manifests feel readable.
  • Practice node maintenance with cordon, drain, and uncordon on a non-critical cluster.
